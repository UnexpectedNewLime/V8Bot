File Purpose
.github/workflows/run-tests.yml Runs pytest and compileall on every push and pull request to every branch
.github/workflows/deploy.yml Builds the Docker image, pushes it to GHCR, and deploys to the production server — triggers only on pushes to main
docker-compose.deploy.yml Production compose file with no build stanza — pulls the pre-built image from GHCR. The deploy workflow copies this to the server on every deploy.
The deploy workflow contains no test job. It relies on branch protection (Step 6) to guarantee that only tested code reaches main, avoiding duplicate test runs and preventing flaky tests from blocking live deployments.
1
Create the deployment directory Server

SSH into the production server and create a working directory for the bot. This is where the compose file, .env, and the SQLite database will live.

mkdir -p ~/v8bot/data

Create the production .env file:

nano ~/v8bot/.env

Minimum required values:

# Required
DISCORD_BOT_TOKEN=your_production_token
DISCORD_GUILD_ID=your_guild_id

The full list of supported settings is documented in .env.example at the repository root. Key optional settings include SCRAPE_INTERVAL_MINUTES, SCRAPER_USER_AGENT, and LOG_LEVEL.

Use a dedicated production Discord bot token — never reuse a development token. Running the same token in two places simultaneously will cause one instance to be disconnected by Discord.
2
Generate a deploy SSH key Server

The Actions runner needs to SSH into the server to copy files and restart the container. Generate a dedicated key pair so it can be revoked independently of your personal keys.

Run these commands on the server:

# Generate key pair with no passphrase
ssh-keygen -t ed25519 -C "github-actions-deploy" -f ~/.ssh/deploy_key -N ""

# Authorise the public key on this server
cat ~/.ssh/deploy_key.pub >> ~/.ssh/authorized_keys

Print the private key — you will paste this into a GitHub secret in Step 4:

cat ~/.ssh/deploy_key

Copy everything from -----BEGIN OPENSSH PRIVATE KEY----- to -----END OPENSSH PRIVATE KEY----- inclusive. Keep this value safe.

3
Create a Tailscale OAuth client Tailscale

The deploy workflow uses the official tailscale/github-action to join the Actions runner to your Tailscale network as an ephemeral node. This allows the runner to reach the server by its Tailscale hostname or IP without exposing SSH to the public internet.

Create the OAuth client:

  1. Open the Tailscale admin console → Settings → OAuth clients.
  2. Click Generate OAuth client.
  3. Set scope to Devices → Write.
  4. Click Generate client.
  5. Copy the Client ID and Client secret. The secret is shown only once.

Authorise the tag:ci tag in your ACL policy:

The workflow tags the ephemeral runner node as tag:ci. Tailscale requires this tag to be declared in the ACL before it can be assigned. In the admin console open Access Controls and add to the tagOwners block:

"tagOwners": {
  "tag:ci": ["autogroup:admin"]
}
The ephemeral node is automatically removed from your Tailscale network when the Actions job finishes, so it will not accumulate in your device list.
4
Add repository secrets GitHub

Go to your repository → Settings → Secrets and variables → Actions and create the following secrets:

Secret name Value
TAILSCALE_OAUTH_CLIENT_ID Client ID from Step 3
TAILSCALE_OAUTH_CLIENT_SECRET Client secret from Step 3
DEPLOY_HOST Tailscale machine name or IP of the server — e.g. my-server or 100.x.x.x. Find this in the Tailscale admin console under Machines.
DEPLOY_USER SSH username on the server, e.g. ubuntu or lime
DEPLOY_SSH_KEY Full private key from ~/.ssh/deploy_key on the server, including the -----BEGIN and -----END lines
GITHUB_TOKEN is provided automatically by GitHub and does not need to be created. It is used by the deploy workflow to authenticate with GHCR when pushing the Docker image.
5
Set GHCR package visibility GitHub

The first time the deploy workflow runs it will push the image to GitHub Container Registry under your account. By default new packages are private, which means the server cannot pull without authentication.

After the first successful push to main:

  1. Go to github.com/UnexpectedNewLime → Packages → v8bot.
  2. Click Package settings (bottom of the sidebar).
  3. Under Danger Zone, set visibility to Public.

This allows the server to pull with a plain docker pull and no credentials.

Prefer to keep the image private? Log in to GHCR on the server once instead. Create a GitHub Personal Access Token (PAT) with the read:packages scope at github.com/settings/tokens, then run:

echo YOUR_PAT | docker login ghcr.io -u UnexpectedNewLime --password-stdin

Docker will store the credential in the server's credential store and use it automatically on subsequent pulls.

6
Block direct pushes and enforce CI on main GitHub

This step does two things: blocks direct pushes to main so that all changes must go through a pull request, and enforces that CI passes before any PR can be merged. Together these form the guarantee that lets the deploy workflow skip running tests.

  1. Go to the repository → Settings → Branches.
  2. Click Add branch protection rule.
  3. Set Branch name pattern to main.
  4. Check Require a pull request before merging. This is what prevents anyone — including you — from pushing commits directly to main. Every change must arrive via a PR.
  5. Check Require status checks to pass before merging.
  6. In the search box that appears, type test and select the test job — this is the job defined in run-tests.yml. It must have run at least once on this repo for it to appear in the list, so open and push a PR first if needed.
  7. Check Require branches to be up to date before merging. This prevents a PR from merging if main has moved forward since the branch was last tested.
  8. Check Block force pushes. This prevents the branch history from being rewritten even by someone with admin access.
  9. Check Do not allow bypassing the above settings. Without this, repository admins can still push directly to main and merge without passing CI.
  10. Click Save changes.
The test status check will not appear in the search list until the CI workflow has run at least once. Open a PR from any branch to trigger it, then come back and complete this step.
Once this rule is saved, even direct pushes from the command line will be rejected with "protected branch hook declined". All changes to main must go through a PR with a passing CI run.

Verifying the setup

Push or merge a change to main and check the following:

GitHub Actions tab

  • Both CI and Deploy workflows should appear.
  • Inside Deploy, you should see the Tailscale connection step, the docker pull, and the docker compose up -d complete without errors.

On the server

# Container should be running
docker ps

# Logs should include "Shard ID None has connected to Gateway"
docker logs car-watch-bot

# Image should show :latest and the commit SHA tag
docker image ls ghcr.io/unexpectednewlime/v8bot

How future deploys work

No further manual steps are needed. Every merge to main follows this path automatically:

  1. A PR branch is pushed — run-tests.yml runs pytest and compileall.
  2. Branch protection blocks the merge until the test job passes.
  3. The PR is merged to main — deploy.yml triggers.
  4. The Docker image is built and pushed to GHCR tagged :latest and :<sha>.
  5. The Actions runner joins your Tailscale network, copies docker-compose.deploy.yml to the server, pulls the new image, and runs docker compose up -d.
  6. The previous container is replaced with zero manual intervention.

The SQLite database at ~/v8bot/data/car_watch_bot.sqlite3 persists across deploys via the volume mount in docker-compose.deploy.yml.
